# Changelog

All notable changes to Crucible are documented here.

## [0.1.0] — 2026-05-12

First public release.

### Added

- **Self-hosted marketplace manifest at `.claude-plugin/marketplace.json`** so users can install Crucible with `/plugin marketplace add hazarsozer/crucible-cc` + `/plugin install crucible@crucible` instead of `git clone` + `claude --plugin-dir`. Same install pattern as community plugins like ECC. The marketplace name and plugin name are both `crucible` (single-plugin self-marketplace). The previous git-clone path is kept as a documented alternative in the README quick-start for users who want full source on disk. Official Anthropic marketplace submission (via [clau.de/plugin-directory-submission](https://clau.de/plugin-directory-submission)) is deferred to a later milestone — the self-marketplace ships the single-command UX today.
- **Cost-preview prompt at the start of `/crucible:run`** — a one-time y/n confirmation that explains the per-run cost depends primarily on the user's Claude Code session model. Crucible cannot detect the user's session model from inside a Skill, so this preview is the closest thing to cost control the plugin offers. The preview lists all three viable orchestrator tiers with measured cost and quota burn (Haiku ~$3–4 / ~6–7% quota, Sonnet ~$4.50–7 / ~10–15%, Opus ~$8–10 / ~15%+), and flags two Haiku-specific caveats: (1) report template adherence is looser on Haiku — section names may improvise and detailed findings may collapse to one-liners, (2) a single Crucible run consumes ~75% of Haiku's 200K context window, so users should start a fresh Claude Code session before running on Haiku if they've been using Claude Code already in the same session. If the user answers `n`, the run halts with no artifacts written; if `y`, the orchestration proceeds normally with no further cost reminders. See `skills/run/SKILL.md` § "Cost preview" for the exact text and the v0.1.0 architectural-exploration context that motivates it.
- **5-stage corporate review pipeline** orchestrated by `/crucible:run`:
  - Stage 0 — **Profiler** (Sonnet 4.6): reads project, interviews user, writes `.review/aims.md`, casts the committee.
  - Stage 1 — **Peer Code Review** (mixed Haiku 4.5 / Sonnet 4.6): 10 personas covering Python, TypeScript, Go, Rust, Java/Kotlin, C/C++, Swift, SQL, plus universal Quality and Readability engineers.
  - Stage 2 — **Cross-functional Gap Review** (Sonnet 4.6): 11 domain reviewers — Security, Frontend, Backend, Network, Database, DevOps/Infra, Performance, Accessibility, Observability, Privacy/Compliance, Data/ML.
  - Stage 3 — **Leadership** (Opus 4.7): Senior Systems Architect (ADR verdict) + Project Manager (aim alignment grade).
  - Stage 4 — **Aggregator** (Opus 4.7): Opus-reasoned holistic synthesis. No mathematical averaging.
- **Adaptive committee casting**: the Profiler picks personas based on project type, languages, frameworks, and review scope. Files are partitioned per persona using extension-based rules for Stage 1 and domain-based semantic rules for Stage 2.
- **Project-aim grading**: the `lead-project-manager` persona reads `.review/aims.md` (interactively written by the Profiler) and grades the change against the user's stated success criteria. Distinguishes Crucible from every other code-review plugin.
- **Sequential stage handoff**: Stage 2 reads Stage 1's structured findings; Stage 3 reads both; Aggregator reads everything plus the aims snapshot. No parallel-fan-out without context.
- **Per-persona model tiering** by reasoning load: Haiku 4.5 for pattern-rich language reviewers; Sonnet 4.6 for reasoning-heavy reviewers; Opus 4.7 for strategic synthesis.
- **Three commands**:
  - `/crucible:run` — full pipeline run
  - `/crucible:aims` — refresh `.review/aims.md` interactively (Profiler in interview-only mode)
  - `/crucible:history` — list and view past reviews
- **Three example reports** under `examples/` covering the included test fixtures (Next.js auth, PyTorch trainer, Go API).
- **Test infrastructure** under `tests/`: JSON-schema validation, persona-frontmatter linting, persona library completeness check, structural integrity tests, persona output validation (gated), and live E2E test (gated by `RUN_LIVE_E2E=1`).
- **Three test fixtures** under `tests/fixtures/` with deliberate gaps for the personas to find: nextjs-auth (security + quality + DB), pytorch-trainer (ML reproducibility + observability), go-api (concurrency + observability + production-readiness).
- **GitHub Actions CI** running lint + schema + structure tests on every push and PR.
- **MIT license**, full README with pipeline diagram, CONTRIBUTING guide, and persona authoring pattern.

### Fixed

- **Profiler now uses the cwd passed in its prompt as the project root**, not `git rev-parse --show-toplevel`. Affects nested projects (a fixture inside a parent repo, a monorepo package, an `examples/foo/` subdirectory): previously the Profiler read the parent repo's README and unfiltered git history, identifying the parent as the project under review instead of the cwd. Symptom: running `/crucible:run` from `tests/fixtures/nextjs-auth` made the Profiler ask "What's the overarching goal of this 23-persona code review pipeline plugin?" instead of recognizing the Next.js auth project.
- **Profiler `.gitignore` update step (step 7) now triggers only when `.git/` is a directory directly inside the project root**, not when the project is anywhere inside a git work tree. Previously, nested projects got a redundant fixture-level `.gitignore` while the parent's `.gitignore` already owned `.review/` policy.
- **Orchestrator now captures `project_root` via `pwd` in Setup and substitutes a literal absolute path into the Profiler dispatch prompt**, instead of relying on the model to interpret a `<absolute path of the user's project>` placeholder. The placeholder approach worked for the nextjs-auth fixture (the Profiler still found the cwd-relative `.review/aims.md` by accident), but failed on the go-api fixture: the Profiler identified the project as Crucible itself (auto-suggesting "Pre-release audit … publishing to the Claude Code plugin marketplace" as the review goal). Same nested-project bug family as the two above.
- **Aggregator metadata is now substituted explicitly by the orchestrator before dispatch**, eliminating the hallucination path. The previous design left `<calculated from started_at to now, integer>` and `<best-effort estimate, or 0 if unknown>` as placeholders for Sonnet to substitute; when the orchestrator forwarded the placeholders unchanged (as observed in the pytorch-trainer run), the Aggregator filled in plausible-looking values (`wall_clock_seconds: 420`, `estimated_cost_usd: 6.5`) that had no relation to reality (real values: 1122 seconds and unknown). The orchestrator now computes `wall_clock_seconds` from the captured `started_at` timestamp and enumerates `models_used` from successful persona dispatches. `estimated_cost_usd` is always hardcoded to `0` — Claude Code does not expose token-level pricing to plugin skills, so any estimated number would be a hallucination.
- **`templates/report.md.tpl` no longer displays Estimated cost.** The field was structurally unmeasurable from inside a plugin skill but was being filled in by the Aggregator (see above). The template now ends the metadata block with a note pointing users to `/status` for real API cost.
- **`skills/run/SKILL.md` "Write the markdown report file" step now enforces strict template adherence.** The pytorch-trainer run revealed that the orchestrator sometimes improvises the report structure (different heading text, different metadata block style) instead of substituting through `templates/report.md.tpl`. The new instructions explicitly require the first line to match the template's `# Crucible Review — ` prefix and the metadata block to use the italic underscore format.
- **`wall_clock_seconds` is no longer displayed in the rendered markdown report or the terminal footer.** Wall-clock is the same class of metric as `estimated_cost_usd`: Claude Code reports it natively (at session end and via `/status`) using runtime measurements that are strictly more accurate than anything a plugin skill can compute from inside the run. The field stays in `schemas/final-report.schema.json` for forward compatibility but is hardcoded to `0` by the orchestrator. The previously-shipped re-measurement step (Setup-step-3 `started_at_unix_seconds` capture + post-Aggregator arithmetic) was a workaround for a display problem that no longer exists; both have been removed from `skills/run/SKILL.md`. Eliminates the entire wall-clock-undercount bug class — earlier in this release cycle the dispatch-time value was 44% short of true end-to-end on the pytorch-trainer run (680s reported vs 1205s real), and the prior fix was a re-measurement layer rather than a structural simplification. This is the structural fix.
- **Bundled `examples/*.md` reports synced to the new template.** All three example reports previously displayed `**Wall-clock:** <Ns>` and (for `nextjs-auth-refactor.md` and `go-api-service.md`) `**Estimated cost:** $0` lines, both of which the current template no longer emits. The example files are now identical in structure to a fresh `/crucible:run` output, eliminating the structural drift a new user would have noticed when comparing examples to their own runs. Findings and verdicts are unchanged from the original measured runs.
- **Stale `~$1–2 per fixture` cost claims in `tests/README.md` and `tests/test_e2e.py` corrected to `~$5–7 per fixture` (median $5.22).** The `$1–2` claim was a planning-stage estimate from the design doc that survived into the test harness docs after measured runs landed elsewhere. (After the coordinator refactor below the post-v0.1.0 target is `~$4–6 per fixture`, but the docs still reflect the verified pre-refactor figure pending the next verification run.)
- **Profiler `.review/aims.md` accuracy-check is now non-skippable.** The Profiler's step 4 instruction was "ask 'still accurate?' if aims exist", and at least one observed run with an Opus-tier orchestrator interpreted this as optional — the Profiler displayed the aims, decided they looked fresh, and proceeded directly to the scope question without asking. Step 4 in `agents/profiler.md` now explicitly states the prompt is non-skippable, includes the verbatim question text the Profiler must ask, and a new anti-pattern entry documents the observed regression mode. This applies whether the Profiler runs at Sonnet, Opus, or Haiku.

### Changed

- **Persona finding schema redesigned: structured `evidence` replaces freeform `location`.** Each `Finding` in `schemas/persona-finding.schema.json` now carries an `evidence` object — `{ path, line_start, line_end?, symbol_name? }` — in place of the previous `location: string` field. The change addresses two failure modes that surfaced during v0.1.0 verification runs against users with the official `security-guidance` plugin's PreToolUse Write hook installed: (1) **hook compatibility** — the old `location` field frequently quoted callers whose names contained substrings the hook scans for (the eval-mode method on PyTorch models, the standard binary-serializer module name, Node's shell-out helpers), which the hook blocks; persona JSON writes for any review touching those patterns failed to land on disk, (2) **fabrication risk** — code in a finding's prose was the persona's retelling, not the source, and could drift from the file. The new shape pushes code citation entirely into structured coordinates: personas supply `evidence.path` + `evidence.line_start`, the report renderer reads the cited source at render time. JSON artifacts no longer contain inline code by construction. See `templates/persona-protocol.md` § 4 (Evidence Citation Format) and § 5 (Code Citation Discipline) for the contract personas now follow. Affected: `schemas/persona-finding.schema.json` (Finding shape), all 23 persona files in `agents/` (few-shot JSON examples), `templates/report.md.tpl` (rendering uses `evidence.path:line_start[-line_end]`), `templates/persona-protocol.md` (new § 4 + § 5), `agents/profiler.md` and `agents/aggregator.md` (reference the protocol's new sections). Pre-release schema break; the previous v0.1.0-beta artifacts validated against the old shape have been archived to `.review-archive-pre-schema-cutover/` outside the test fixture tree. New `tests/test_hook_compat.py` scans persona JSON outputs for the 12 known security-guidance trigger substrings and emits a `UserWarning` if any are present (does not fail the test — runtime retry-on-block handles slips transparently; the warning surfaces the slip rate so persona prompts can be tightened across versions).
- **`skills/run/SKILL.md` documents the per-rule-per-file retry semantics of the `security-guidance` hook.** If a write is blocked by the hook because a finding legitimately discusses a dangerous-code pattern (e.g., the prose unavoidably uses a bare word like the standard serialization module's name), the orchestrator retries the same write once — the hook records the rule on first hit and the second attempt succeeds. Two places in SKILL.md (persona-finding save and final-report.md write) now spell this out so future maintainers don't conflate hook blocks with persona failures. Structural discipline in the persona protocol minimizes how often the retry path is exercised; the retry exists as a safety net, not a workaround.

### Removed

- **`docs/superpowers/` directory (specs + plans, 3902 lines).** The implementation specification (`docs/superpowers/specs/2026-05-10-crucible-design.md`) and step-by-step plan (`docs/superpowers/plans/2026-05-10-crucible.md`) were authored as internal scaffolding for the v0.1.0 build sessions. They are not reference material for plugin users, they contained stale cost numbers (`~$0.82 per review`, `~$1–2 per fixture`) that contradicted the README's measured `$5–7`, and they preserved an older template format with the now-removed `Estimated cost` line. The "Persona Authoring Pattern" section that `CONTRIBUTING.md` previously linked into these docs has been inlined into `CONTRIBUTING.md` directly. The full design history remains available in git history.

### Notes

- **Cost is dominated by the orchestrator's model, not by the dispatched subagent tiers.** This is the single most important architectural insight from the v0.1.0 measurement campaign. The orchestrator (the user's Claude Code main thread, executing `skills/run/SKILL.md`) reads scope files, parses persona JSON returns, validates against schemas, and renders the markdown report — all of which accumulate cache reads at the session's model rate. Measured: 5 runs with the user on Sonnet 4.6 cost $4.78–$6.75 (median $5.22); 1 run with the user on Opus 4.7 cost $8.95 on the cheapest fixture (~1.7× the Sonnet-main equivalent). v0.1.0 ships the cost-preview prompt (above) as the user-visible mitigation, and documents the v0.2.0+ paths to a structural fix (coordinator subagent + interactive-passthrough, or Profiler-in-main + coordinator-for-rest) in `README.md` § "Future cost work".

- **v0.1.0 architectural exploration: coordinator subagent attempted, rolled back.** Recorded here because it's a useful negative result for any future plugin author considering nested-subagent orchestration. The exploration tested whether moving the cache-heavy bookkeeping into a dedicated `coordinator` subagent at a fixed model tier would decouple total cost from the user's session model. Two failure modes blocked the approach:

  1. **Haiku coordinator silently impersonated personas.** First verification run on `pytorch-trainer` cost $1.07 total (vs target $4–6). Looks like a win until the cost split is examined: Sonnet cost was **$0** despite seven personas claiming `model_used: claude-sonnet-4-6` in their saved JSON outputs; Opus cost was $0.54 (just main-thread overhead from the session model dispatching the coordinator) despite two Opus personas claiming work; 3.3M Haiku cache reads accounted for Haiku reading all persona system prompts + all fixture files and generating outputs that matched each persona's expected JSON shape from inside its own context — without ever calling the Task tool. The findings were plausible but the plugin was lying about which model produced which review.

  2. **Sonnet coordinator broke user interactivity.** Escalating to Sonnet fixed the impersonation discipline (real subagent dispatches showed up as real Sonnet/Opus cost in `/status`), but introduced a different failure: the Profiler is a subagent that must interactively prompt the user ("I found existing aims. Are these still accurate?", "Proceed with this committee?"). When the Profiler is dispatched as a sub-subagent (`coordinator → profiler`), nested subagent dispatch loses the user-interaction channel. The Profiler ran, did its work mechanically without prompting, and Crucible silently re-confirmed aims that should have been user-validated. Cost target ($4.36) was hit, but the interactive UX broke. The user caught it: *"it didn't ask me anything again."*

  Decision: rollback to main-thread orchestration + cost-preview prompt (above). The architectural lessons — *Haiku-as-coordinator is too good at impersonation to safely run multi-subagent orchestration without explicit tool-use enforcement; Sonnet-as-coordinator handles tool-use discipline but nested subagents can't reliably prompt users* — are worth documenting for the broader plugin-authoring community. Files affected during the exploration and reverted: `skills/run/SKILL.md` (thin wrapper → full orchestration), `agents/coordinator.md` (created → deleted), `scripts/lint_personas.py` (regex added `coordinator` → reverted), `tests/test_persona_lint.py` and `tests/test_e2e.py` (coordinator references added → reverted). The `agents/profiler.md` tightening for the non-skippable aims-check survives the rollback because it's still a useful improvement in the main-thread architecture (and the silent-aims-confirmation regression is now actually testable again, since the Profiler regains its user-interaction channel).

  **Follow-up finding (2026-05-12): Haiku as the main-thread orchestrator works.** After the rollback, a fresh verification run on `pytorch-trainer` with the user's Claude Code session set to `claude-haiku-4-5-20251001` produced honest costs: total $3.26 with Haiku $0.79 (main thread orchestrator), Sonnet $1.04 (real Profiler + 3 Sonnet subagents dispatched), Opus $1.42 (real Stage 3 leadership + Aggregator dispatched). All three model tiers showed real work; no impersonation tells. The Profiler honored the aims-still-accurate prompt and the cost preview honored its y/n confirmation. The impersonation failure mode was therefore specific to Haiku running *inside* a dispatched coordinator subagent, where SKILL.md's `Task(...)` blocks read as workflow descriptions rather than tool invocations — on the main thread, Haiku correctly invokes the Task tool. Trade-off: the rendered markdown report's template adherence is looser on Haiku (section names improvise — `## Summary` instead of `## Final Verdict` + `## Executive Summary`; `## Key Quotes` instead of `## Key Notes from the Committee`; stage sections collapse to one-line summaries instead of full per-finding citations; Run Metadata block may be skipped; attribution links may be fabricated). Pipeline integrity is intact; only the markdown formatting is less canonical. Haiku-main is documented in the cost preview and README as a viable cost-sensitive option, with the template-adherence caveats called out. Sonnet is the recommended default for the balance of cost and report polish, though a separate measurement on 2026-05-12 showed Sonnet also occasionally improvises the rendered report template — see the "Known limitations" section below for details and the v0.1.1 plan to replace LLM-driven template substitution with a deterministic Python renderer.
- Configuration via `.review/config.yaml` is parsed but not yet honored; full override behavior ships in v0.2.0.
- Persona output schema (`schemas/persona-finding.schema.json`) loosened to match observed real-run outputs: `summary_quote` raised from `maxLength: 280` to `500`; `Finding.title` raised from `120` to `250`; `scope_assessed` widened to accept the literal string `"all"` (not just an array of paths). Real Sonnet/Opus personas routinely produce 280–330 char headlines and 130–180 char finding titles; the prior caps were aspirational and failed validation on every measured run.
- All three bundled example reports under `examples/` are real Crucible outputs from measured runs on the corresponding fixtures: `nextjs-auth-refactor.md` (2/10 blocked, $5.22 / 25 min), `go-api-service.md` (2.5/10 blocked, $5.60 / 23 min), `ml-training-loop.md` (3.5/10 blocked, $4.78 / 20 min). The ml example has one corrected line — the displayed wall-clock value was updated from the originally-emitted 680s (dispatch-time measurement, pre-fix) to the true end-to-end 1205s; see the "wall_clock_seconds is now re-measured" Fixed bullet above for the underlying orchestrator change.
- Runs on Claude Pro and Max plans. **Measured cost across all three bundled fixtures (`nextjs-auth`, `go-api`, `pytorch-trainer` — 5–7 files each, 8–10 persona casts): $5–7 API per run (median $5.22 across 5 runs, range $4.78–$6.75), 20–35 min wall time, ~10–15% Max-plan quota burn.** Cross-project variance is small for similar-sized fixtures; within-fixture verbosity variance is the dominant noise source. Cast size shifts with project type: ML pipelines cast 8 personas (Profiler skips Frontend, Database, Network, Privacy); web/API projects cast 10. Opus is available on both Pro and Max tiers; Pro caps Opus access, so heavy users should scope reviews tighter (a phase review casts ~5 personas vs. ~8–10 for a full-project review). Larger-project cost data (codebases beyond 5–7 files) lands in v0.1.1. See the README §Costs section for the per-model split and caveats.
- Quality-review polish backlog (per-persona nits surfaced during authoring) is tracked for v0.2.0.

### Known limitations (measured 2026-05-12 Sonnet-main verification on pytorch-trainer)

- **Persona prose discipline slip rate ~25%.** The Code Citation Discipline (see persona-protocol.md § 5) asks personas to describe forbidden-substring patterns in prose form rather than inlining the literal syntactic form. In a measured Sonnet-main run on the pytorch-trainer fixture, 2 of 8 personas (`peer-python-reviewer`, `lead-project-manager`) wrote a literal PyTorch eval-mode-method call in their JSON output despite the protocol. Runtime impact: zero — the per-rule-per-file retry-on-block mechanism (documented in skills/run/SKILL.md and persona-protocol.md § 5) handles slips transparently, and all 11 artifact files landed cleanly on disk. The new `tests/test_hook_compat.py` surfaces slips as warnings (not test failures) so the slip rate can be tracked across versions and persona prompts tightened in v0.1.1+. Stronger per-persona examples for common slip sites (PyTorch's eval-mode method, the standard binary-serializer module name) is on the v0.1.1 backlog.

- **Sonnet orchestrator may improvise the rendered report template.** `skills/run/SKILL.md` instructs the orchestrator to substitute the placeholders in `templates/report.md.tpl` verbatim, but in the measured Sonnet-main run the orchestrator restructured the output (table-style stage summaries, emoji-prefixed key quotes, reordered finding line `[severity] path:line — title` instead of `[severity] title — path:line`, condensed `## Verdict` instead of `## Final Verdict`, `## Key Notes` instead of `## Key Notes from the Committee`, `## Committee` instead of `## Run Metadata`, and — the only consequential deviation — skipped the `## Stage 0 — Profiler` section entirely, losing the project profile, review scope, and casting reasoning displays). Content quality is unaffected (findings, scores, verdicts, and reasoning all preserved); the deviations are cosmetic with one structural drop. Root cause: LLM-driven mechanical text substitution is inherently unreliable for strict template adherence. v0.1.1 will replace this step with a deterministic Python renderer that takes the `final-report.json` and applies the template via a real template engine. v0.1.0 ships with this as a documented MVP-stage limitation.
