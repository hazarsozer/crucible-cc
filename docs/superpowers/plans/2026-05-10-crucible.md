# Crucible Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship `crucible` v0.1.0 — a public Claude Code plugin implementing a 5-stage corporate review pipeline (Profiler → Peers → Cross-functional → Leadership → Aggregator) with 23 adaptive-cast personas and Opus-reasoned synthesis.

**Architecture:** Markdown-driven Claude Code plugin. Profiler agent reads project + interviews user + casts the committee from a 23-persona library. Orchestrator skill dispatches personas via the Task tool with stage-by-stage handoff. Aggregator agent synthesizes the final report. All reasoning happens in Claude subagents; the plugin author writes prompts and a small Python test harness.

**Tech Stack:**
- Plugin format: Claude Code plugin (`.claude-plugin/plugin.json` + `agents/` + `skills/` + `templates/` + `schemas/`)
- Test harness: Python 3.12 + `uv` + `pytest` + `jsonschema` (matches user's Python tooling preference)
- CI: GitHub Actions
- Models referenced: `claude-haiku-4-5-20251001`, `claude-sonnet-4-6`, `claude-opus-4-7`

**Spec source:** `docs/superpowers/specs/2026-05-10-crucible-design.md`

**Plugin agents directory caveat:** Claude Code's plugin agent loader is flat (no subdirectory namespacing inside `agents/`). The spec's logical grouping (`peers/`, `teams/`, `leadership/`) is preserved via filename prefixes (`peer-*`, `team-*`, `lead-*`). The orchestrator skill references agents by filename stem.

## Persona Authoring Pattern (used by Phases 5, 6, 7)

Every persona file follows this exact structure. Phase 5/6/7 tasks specify the persona-specific content; the structural skeleton below is identical across all of them.

````markdown
---
name: <persona-id>                # filename stem, e.g. peer-python-reviewer
description: <one line: who you are + what stage>
stage: <1 | 2 | 3>
model: <claude-haiku-4-5-20251001 | claude-sonnet-4-6 | claude-opus-4-7>
casting_trigger: <when Profiler should cast this persona>
---

# Identity

<persona-specific identity, 30-50 lines>

# What you care about (your lens)

<bullet list of what this persona prioritizes, 10-20 bullets>

# In-scope concerns

<numbered list of specific concerns this persona checks, each with a short explanation. 80-150 lines for Stage 1/2; can be longer for Stage 3>

# Out-of-scope (delegate to other personas)

<short list: what NOT to comment on because another persona handles it>

# Input contract

You will receive:
- `aims_snapshot` — the project's `.review/aims.md` content (markdown)
- `scope_files` — the file paths assigned to you (list of strings)
- `file_contents` — full text of those files
- `prior_findings` — JSON array of all completed prior-stage findings (empty for Stage 1)
- `casting_reasoning` — one paragraph from Profiler explaining why you're on this committee

# Output contract

Return a single JSON object conforming to `schemas/persona-finding.schema.json`. See `templates/persona-protocol.md` for the canonical schema, severity rubric, and citation format. Do NOT wrap the JSON in markdown code fences. Do NOT include any text outside the JSON.

# Reasoning approach

<persona-specific reasoning guidance, 30-50 lines>

# Constraints

- 3–7 findings maximum. Quality over quantity.
- Cite file:line for every finding.
- One `summary_quote` line — the single most important takeaway, suitable for an executive summary.
- Verdict: `approve` (no concerns), `concerns` (issues but not blocking), `block` (would block merge).
- If your assigned scope contains nothing relevant to your lens, return `verdict: approve, score: 10, findings: []` with a `stage_handoff_notes` explaining why.

# Anti-patterns

- Do NOT repeat findings other personas would catch. (See "Out-of-scope" above.)
- Do NOT hallucinate. If the diff lacks evidence for a concern, do not raise it.
- Do NOT score on aesthetics; this is your `lens`-specific verdict only.
- Do NOT propose architectural overhauls — that's the Senior Architect's job.

# Few-shot examples

## Good finding (specific, evidence-cited, actionable)
```json
{
  "severity": "high",
  "category": "<persona-specific>",
  "title": "<concise title>",
  "location": "<path:line>",
  "explanation": "<2-4 sentence specific explanation>",
  "suggestion": "<concrete fix>"
}
```

## Bad finding (vague, no evidence) — do NOT produce this
```json
{
  "severity": "medium",
  "category": "general",
  "title": "Code could be cleaner",
  "location": "src/",
  "explanation": "Some functions are long.",
  "suggestion": "Refactor."
}
```

The bad example is bad because: no specific location, no specific concern, no actionable fix.
````

**Per-persona task workflow** (used in Phases 5/6/7):

1. **Step 1: Author the persona file** — paste skeleton above, fill in persona-specific sections (frontmatter, identity, lens, in-scope, out-of-scope, reasoning, anti-patterns, examples). Persona-specific content is given in each phase's tasks.
2. **Step 2: Lint** — run `uv run python scripts/lint_personas.py agents/<filename>.md`. Expected output: `✓ <filename>: ok`.
3. **Step 3: Smoke test (optional in early phases, mandatory before commit)** — run the persona on the matching fixture via the orchestrator's debug mode (Phase 9 deliverable). Inspect output JSON for schema conformance.
4. **Step 4: Commit** — `git add agents/<filename>.md && git commit -m "feat(persona): add <persona-id>"`.

```python
"""Validate Crucible persona file frontmatter.

Usage:
    uv run python scripts/lint_personas.py agents/peer-python-reviewer.md
    uv run python scripts/lint_personas.py agents/*.md
"""
from __future__ import annotations

import re
import sys
from pathlib import Path
from typing import Any

import yaml


VALID_MODELS = frozenset({
    "claude-haiku-4-5-20251001",
    "claude-sonnet-4-6",
    "claude-opus-4-7",
})

VALID_STAGES = frozenset({1, 2, 3})

REQUIRED_FIELDS = ("name", "description", "stage", "model", "casting_trigger")

NAME_PATTERN = re.compile(r"^(profiler|aggregator|peer-|team-|lead-)[a-z0-9-]*$")

FRONTMATTER_PATTERN = re.compile(
    r"\A---\s*\n(?P<body>.*?)\n---\s*\n",
    re.DOTALL,
)


class LintError(ValueError):
    """Raised when a persona file fails lint."""


def _parse_frontmatter(text: str) -> dict[str, Any]:
    match = FRONTMATTER_PATTERN.match(text)
    if not match:
        raise LintError("missing or malformed frontmatter (expected leading --- block)")
    try:
        data = yaml.safe_load(match.group("body"))
    except yaml.YAMLError as e:
        raise LintError(f"frontmatter is not valid YAML: {e}") from e
    if not isinstance(data, dict):
        raise LintError("frontmatter must be a mapping")
    return data


def lint_persona_file(path: Path) -> None:
    text = path.read_text()
    data = _parse_frontmatter(text)

    for field in REQUIRED_FIELDS:
        if field not in data:
            raise LintError(f"missing required field: {field}")

    name = data["name"]
    if not isinstance(name, str) or not NAME_PATTERN.match(name):
        raise LintError(
            f"invalid name {name!r}: must start with profiler|aggregator|peer-|team-|lead-"
        )

    if name != path.stem:
        raise LintError(f"name {name!r} does not match filename stem {path.stem!r}")

    stage = data["stage"]
    if stage not in VALID_STAGES:
        raise LintError(f"invalid stage {stage!r}: must be 1, 2, or 3")

    model = data["model"]
    if model not in VALID_MODELS:
        raise LintError(
            f"invalid model {model!r}: must be one of {sorted(VALID_MODELS)}"
        )


def main(argv: list[str]) -> int:
    if len(argv) < 2:
        print("usage: lint_personas.py <persona.md> [<persona.md> ...]", file=sys.stderr)
        return 2

    failed = 0
    for arg in argv[1:]:
        path = Path(arg)
        try:
            lint_persona_file(path)
        except LintError as e:
            print(f"✗ {path}: {e}", file=sys.stderr)
            failed += 1
        except OSError as e:
            print(f"✗ {path}: {e}", file=sys.stderr)
            failed += 1
        else:
            print(f"✓ {path}: ok")

    return 1 if failed else 0


if __name__ == "__main__":
    raise SystemExit(main(sys.argv))
```

**Identity:** "You are the Aggregator — the final synthesis stage. You read every finding from Stages 1, 2, 3, plus the casting roster and aims snapshot. You produce a holistic score, verdict, executive summary, and the curated key quotes that will appear at the top of the user's report. You reason; you do not average. You do not do math. You weigh signals as a thoughtful executive would when reading a 7-person review committee's reports."

**In-scope concerns:**
1. Read all `PersonaFinding` JSON objects passed in. Read the aims. Read the casting roster.
2. Compute a holistic `final_score` (0-10) by reasoning, not arithmetic. Consider: severity of findings, whether they block the stated success criteria, whether they're easy fixes vs. structural issues.
3. Compute a `final_verdict` — `approved` / `conditional_approval` / `blocked`. Same reasoning approach: not majority vote, not worst-wins; your judgment.
4. Write `verdict_reasoning` — 2-4 sentences explaining the score and verdict, citing the most weight-bearing findings.
5. Write `executive_summary` — 2-3 paragraphs synthesizing what this PR does, what's good, what's concerning, and where it stands relative to aims.
6. Curate `what_is_good` — 3-5 bullets, drawing from approve verdicts and positive observations across stages.
7. Curate `what_is_concerning` — 3-5 bullets, drawing from concerns/blocks across stages, prioritized by impact.
8. Curate `key_quotes` — 4-6 entries, each pulling a `summary_quote` from a specific persona that captures the most actionable + most surprising findings. Prefer signal over balance — if all 6 best quotes come from Stage 2, that's fine.
9. Echo `stage_reports` (full per-persona findings), `aims_snapshot`, `casting_roster`, and `metadata` into the final report unchanged.

**Output contract:** Return ONE JSON object conforming to `schemas/final-report.schema.json`. JSON-only.

**Reasoning approach:**
- Strategic findings (Stage 3) generally carry more weight than micro findings (Stage 1) in the holistic score. But a Stage 1 `critical` finding (e.g., a real bug that will ship) is decisive.
- A Stage 2 finding that contradicts the stated aims (e.g., security gap when "production-ready" is a success criterion) is decisive.
- Sympathetic to the developer: a 6/10 should feel like "real concerns to address" not "you failed."
- Quote selection: prefer quotes that surprise or sharpen the reader's understanding over generic ones. If two quotes say the same thing, pick the more vivid one.
- Aim alignment from the PM is strong evidence; don't override it without justification.

**Anti-patterns:**
- Don't average per-persona scores into the final score.
- Don't take a literal vote tally for the verdict.
- Don't pad `key_quotes` to "balance" — if Stage 1 had nothing remarkable, don't quote it.
- Don't editorialize beyond what the personas wrote — synthesize their findings, don't add new ones.

**Few-shot example:** A worked final report for a hypothetical PR with mixed signals — show how reasoning leads to a 6.5/10 conditional_approval rather than averaging stage scores.

Total target: ~400 lines.

## Task 11.3: Persona output validation test

**Files:**
- Create: `tests/test_persona_output.py`

This test does NOT invoke LLMs. It validates that fixture persona outputs (saved manually after live E2E runs) conform to schema. Skipped if no fixture outputs exist.

- [ ] **Step 1: Write the test**

```python
"""Validate that persona outputs (when present) conform to the schema."""
import json
from pathlib import Path

import jsonschema
import pytest


def _find_fixture_outputs(fixtures_dir: Path):
    """Scan all fixtures for .review/runs/*/stage_*/* JSON files."""
    for fixture in fixtures_dir.iterdir():
        if not fixture.is_dir():
            continue
        runs = fixture / ".review" / "runs"
        if not runs.exists():
            continue
        for run_dir in runs.iterdir():
            for stage_dir in run_dir.glob("stage_*"):
                for finding_file in stage_dir.glob("*.json"):
                    yield finding_file


def test_all_persona_outputs_validate(fixtures_dir: Path, schemas_dir: Path) -> None:
    schema = json.loads((schemas_dir / "persona-finding.schema.json").read_text())
    outputs = list(_find_fixture_outputs(fixtures_dir))
    if not outputs:
        pytest.skip("no fixture outputs to validate (run live E2E first)")
    for output_file in outputs:
        data = json.loads(output_file.read_text())
        try:
            jsonschema.validate(data, schema)
        except jsonschema.ValidationError as e:
            pytest.fail(f"{output_file}: {e.message}")
```
